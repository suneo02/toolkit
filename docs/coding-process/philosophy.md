# Agent-Native 开发哲学与原则

本文件定义从“静态 Spec 驱动”到“Agent-Native 协作”的核心理念，用于统一团队心智与决策方式。

## 一、为什么要从静态 Spec 转向契约与活文档

传统的长 Spec 容易成为执行的枷锁：
- 锚定效应：早期方案被固化，后续探索发现也难以调整。
- 过早承诺：没有证据就写死方案，进入实现后才发现入口/依赖不匹配。
- 版本漂移：代码库和依赖变化快，静态文档易过时。

因此，Spec 应“瘦身为契约”，实现细节交给探索与验证。

## 二、核心原则

### 1. Contract-first
只写“不会轻易过时的内容”：
- 目标与非目标
- 用户可见行为
- 约束（兼容性、性能、类型安全等）
- 验收标准
- 关键风险与注意事项

### 2. Living docs
文档不是一次性产物，而是执行过程中持续更新的记录：
- todo 会随着探索结果更新
- 方案选择可以被新证据修正

### 3. Parallel explore
并行获取事实与证据，避免“纸上谈兵”：
- explore: 在仓库中找入口、调用链、影响面
- librarian: 查外部最佳实践与官方文档
- oracle: 复杂取舍时审架构与风险

### 4. Verify early
每个 todo 必须绑定验证方式，避免最后一起发现问题。

### 5. Review by subagent
代码评审优先交给专业子代理（如 Oracle），而不是维护独立 review skill：
- 子代理有独立上下文，更专注且不污染主流程
- 模型可配置高推理版本，深度与质量更稳定
- 可与验证阶段融合，形成自动 review 循环

## 三、Spec 的新定位

Spec 不消失，但角色改变：
- 过去：定义“怎么做”并写死步骤
- 现在：记录“意图与约束”，把“怎么做”交给探索与验证

建议将 Spec 拆成两层：
- Contract Spec: 意图/约束/验收/风险
- Execution Plan: todo + 验证方式 + 当前影响面

## 四、适用场景与边界

需要保留契约型 Spec 的场景：
- 跨团队协作与对齐
- 高风险改动（安全/权限/支付/数据一致性）
- 需求易变但需要边界锁定

不建议写死实现细节的场景：
- 入口/依赖不明确
- 代码库在迁移中
- 方案可能需要多轮探索验证
