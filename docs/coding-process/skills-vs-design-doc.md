# AI Coding 中的 Agent、Sub-Agent、Skills 与设计文档

本文档整合比较 AI 编程领域中的四个核心概念，帮助开发者理解它们的关系并做出合理取舍。

---

## 1. 核心概念定义

### Agent（主代理）

Agent 是一个能够根据提示自动推理、生成、执行任务的 AI 系统。它拥有通用理解和行动能力，在软件开发中相当于一个"智能助手"。

**主要职责**：
- 理解全局意图，负责决策和任务分配
- 维护整个对话的长短期记忆
- 回答编程问题、自动写代码、自动修复、方案设计与重构建议

### Sub-Agent（子代理）

Sub-Agent 是独立于主 Agent 的小型 AI 代理，专门用来处理特定任务。它可以拥有自己的上下文、指令、工具权限等。

**主要职责**：
- 处理特定子任务（如：只负责写单测、只负责查文档）
- 仅拥有当前子任务相关的局部上下文，防止干扰
- 像给主 Agent "雇佣一个专家"

**典型应用**：
- 代码审查 Sub-Agent
- 安全审计 Sub-Agent
- 性能分析 Sub-Agent

### Agent Skills（技能）

Agent Skills 是一种给 AI Agent "装插件"的机制，相当于模块化的能力集或工作流说明书。通常是一个 **SKILL.md + 相关资源文件夹**，定义某类任务该怎么做、用什么步骤、注意事项等。

**主要特点**：
- 执行具体动作（如：读文件、运行 Terminal 命令、搜索网页）
- 每次调用是原子性的，通常无记忆
- 可动态加载和按需执行，不会浪费上下文空间

### 设计文档（Design Document）

设计文档是人类开发过程中用于定义系统、API、架构、规范的文字说明。它是 AI 编程的"锚点"，确保开发不偏离轨道。

**主要作用**：
- 定义系统边界、逻辑、接口规范
- 提供 Skills 和 Sub-Agent 的规范参考
- 确保团队共识、架构稳定性和长期可维护性

---

## 2. 角色类比理解

| 概念 | 角色类比 | 说明 |
|------|---------|------|
| **Agent** | 总指挥 / 首席程序员 | 理解全局意图，负责决策和任务分配 |
| **Sub-Agent** | 专项小组 / 实习生 | 处理特定子任务，拥有局部上下文 |
| **Agent Skills** | 程序员的手（工具箱） | 执行具体动作，原子性调用 |
| **设计文档** | 架构图 / 开发准则 | 静态的、高维度的参考信息 |

## 3. 优劣势对比

### 综合评分对比

| 维度 | Agent | Agent Skills | Sub-Agent | 设计文档 |
|------|-------|--------------|-----------|---------|
| 自动执行能力 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐ |
| 可复用性 | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 专业性/细分任务能力 | ⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 设计复杂度 | 简单 | 中等 | 高 | 中等 |
| 上手难度 | 低 | 中等 | 高 | 低 |
| 协作/标准化 | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

### 详细优劣分析

#### Agent（单一智能体）

| 优势 | 劣势 |
|------|------|
| 开发和部署最简单 | "上下文迷失"：代码量大或任务复杂时容易忘记之前的决策 |
| 上下文连贯性强，不需要在多个实体间同步信息 | Token 消耗随对话增长呈指数级上升 |
| 通用性强，不依赖特定场景 | 单代理处理复杂任务时容易出错或超时 |
| 交互式，即时反馈 | 扩展难，若任务分解不当效率低下 |

#### Sub-Agent（子智能体架构）

| 优势 | 劣势 |
|------|------|
| 上下文隔离：主智能体的上下文保持整洁 | 通讯开销大（Token 翻倍） |
| 可并行处理多任务 | 如果主 Agent 任务下达不明确，子智能体容易"南辕北辙" |
| 专注性高，适合专业任务执行 | 管理复杂，需设计工作流 |
| 独立性强，减少干扰 | 多代理调用 API 费用增加 |

#### Agent Skills（原子技能）

| 优势 | 劣势 |
|------|------|
| 精准、确定、可复用 | 技能越多，Agent 选择技能时的推理压力越大 |
| 高可复用性：多个项目或多个代理都能重复使用 | 需要额外设计技能规范并维护相关文件和脚本 |
| 可维护性：独立维护技能逻辑 | 如果设计不清晰可能无法被 Agent 精准调用 |
| 节省上下文：采用分阶段加载 | 灵活性有限，适合固定功能 |

#### 设计文档

| 优势 | 劣势 |
|------|------|
| 约束与对齐：AI 不会随机改写接口定义 | 编写和维护需要人力成本 |
| 清晰沟通：开发者/Agent/团队之间信息一致 | 没有自动化能力，本身不会执行任务 |
| 可追踪规范变更，便于长期维护 | 如果文档与实际代码脱节，反而会误导 Agent |
| 减少 AI 幻觉，提升输出质量 | 静态性，不适应快速变化的需求 |

## 4. 场景决策指南

### 场景 A：修复一个小 Bug 或添加简单函数

**选择**：Agent + Agent Skills

**策略**：不需要子智能体。直接给 Agent 读写文件和运行测试的技能（Skills），让它在单一上下文内快速完成并自我验证。

### 场景 B：重构大型模块或开发一个全新功能

**选择**：设计文档 + Agent + Sub-agents

**策略**：
1. **先写设计文档**：明确 API 协议和数据流
2. **主 Agent 领衔**：读取文档并制定 Plan
3. **分发给 Sub-agents**：例如，Sub-agent A 负责修改逻辑代码，Sub-agent B 同时根据文档编写测试用例

### 场景 C：多语言、多平台协作（如前端 + 后端同步修改）

**选择**：多 Sub-agents + 共享设计文档

**策略**：创建专门针对 Python 后端和 React 前端的子智能体，它们分别只关注各自领域的文件。通过**设计文档**作为它们沟通的唯一事实来源，防止前端改了 API 而后端不知道。

### 按任务复杂度取舍

| 任务复杂度 | 推荐组合 | 说明 |
|-----------|---------|------|
| **简单任务**（<1小时） | Agent + Skills + 简短设计文档 | 速度快，舍弃 Sub Agent 以避免复杂 |
| **中等任务** | Agent + Sub Agent（少量）+ 详细设计文档 | 平衡效率和准确 |
| **复杂任务**（大规模编码） | 多代理（Agent + Sub Agent）+ Skills + 设计文档 | 协作强，但需监控开销 |

## 5. 结合使用策略

### 将设计文档转化为 Agent Skills 的输入

把设计文档放入 agent skills 的文件夹中，作为"资源"部分。SKILL.md 文件可以引用文档的关键章节。

**示例结构**：

```
detail-page-validation/
├── SKILL.md          # 描述技能目的和检查点列表
├── instructions/     # 存放设计文档的摘要或关键 snippet
├── scripts/          # 包含自动化脚本（如 diff 代码变更并匹配文档要求）
└── resources/        # 完整设计文档文件
```

**SKILL.md 示例内容**：
- 描述技能目的（如"基于设计文档验证详情页改动，包括数据一致性、UI 渲染和错误处理"）
- 列出检查点

### 同步机制

- **版本控制**：使用 Git，文档更新时触发 agent skills 的重新加载或测试
- **定期审计**：每周 review AI 生成的输出是否符合文档，避免漂移

### 工具集成

- 如果使用 Claude API 或 Claude Code，agent skills 可以与代码执行环境结合（如运行 bash 命令检查页面渲染）
- 结合 CI/CD 管道中嵌入 AI 技能，自动运行检查点测试

## 6. 总结性建议

### 三条核心原则

1. **技能（Skills）是基础**：无论你用什么架构，都要先打磨好 Agent 的基本功（如搜索代码、修改文件、运行测试、Linter 检查）

2. **文档是灵魂**：AI 编程中，**"Prompt 并不等于文档"**。把复杂的逻辑写在设计文档里（Markdown 格式），作为 Context 喂给 AI，其效果远好于在对话中反复叮嘱

3. **子智能体（Sub-agent）是高级调优**：当你的 Agent 总是因为对话太长而"变笨"时，才是引入 Sub-agent 的最佳时机。不要过早优化，因为多智能体系统的调试复杂度非常高

### 取舍总结

| 场景 | 推荐方案 |
|------|---------|
| 个人开发者或中小项目 | Agent + Skills |
| 大型项目或团队 | Sub-Agent + Skills + 完善的设计文档 |
| 永远不能省略的 | **设计文档** — 它是让 AI "正确行动"的基础 |

### 实施步骤

1. **评估当前文档**：列出所有检查点，标记高风险区域
2. **构建 1-2 个简单 agent skills**：从"校验"开始，测试在小改动上
3. **集成到开发流程**：如在 VS Code 或 IDE 中用 Claude 插件加载技能
4. **监控效果**：追踪 bug 率和开发时间，调整比例
5. **风险缓解**：定期人类 review AI 输出，确保技能不引入新 bug

### 经验提示

- 从单代理开始测试，若错误率 >20%，引入 Sub Agent
- 始终用设计文档作为"锚点"，因为 AI Coding 的瓶颈往往是输入清晰度而非工具本身
- 好的 Spec 可以提高 50% 生产力，减少 30-50% 迭代轮次
