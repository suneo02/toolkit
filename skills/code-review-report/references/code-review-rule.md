# 代码审查规则（精简版）

## 目标

- 审查当前分支相对基准分支的提交。
- 只输出一份 Markdown 报告，聚焦问题与风险。

## 审查维度

1. 需求实现完整性
   - 通过：需求点全部覆盖且实现一致。
   - 不通过：遗漏、偏差或破坏既有功能。
2. 线上风险评估
   - 通过：稳定性/性能/安全/一致性无明显风险。
   - 不通过：存在可复现风险或隐患。
3. 变更范围控制
   - 通过：变更仅限需求相关。
   - 不通过：包含无关改动。
4. 代码规范符合性
   - 通过：遵循规范与架构约定。
   - 不通过：违反规范或明显坏味道。

## 输出规则

- 仅列出问题；无问题写“未发现问题”。
- 每条问题必须包含：文件/行、影响、建议。
- 风险类问题必须标注严重级别（高/中/低）。
